1. Wstęp

Cel aplikacji

ExamIQ to interaktywna aplikacja webowa stworzona z myślą o uczniach technikum informatycznego przygotowujących się do egzaminu zawodowego INF.04 – Projektowanie, programowanie i testowanie aplikacji. Głównym celem projektu jest dostarczenie uczniowi nowoczesnego, angażującego narzędzia do systematycznej i efektywnej nauki, które łączy klasyczne podejście do quizów z elementami grywalizacji oraz oceną kodu przez sztuczną inteligencję.

Dla kogo jest przeznaczona

Aplikacja skierowana jest przede wszystkim do uczniów pięcioletnich techników o profilu programistycznym, którzy przystępują do egzaminu potwierdzającego kwalifikację INF.04. Może być również pomocna dla osób uzupełniających wykształcenie zawodowe, nauczycieli chcących demonstrować materiał na lekcjach oraz wszystkich zainteresowanych utrwaleniem wiedzy z zakresu programowania, baz danych i administrowania systemami.

Co umożliwia aplikacja

ExamIQ oferuje trzy główne tryby pracy. Tryb nauki pozwala na rozwiązywanie pytań testowych (zamkniętych i otwartych) z natychmiastową informacją zwrotną, konfigurowalne limity czasowe oraz system serii poprawnych odpowiedzi motywujący do regularnej nauki. Tryb egzaminu symuluje warunki rzeczywistego egzaminu zawodowego — użytkownik rozwiązuje zadania w określonym czasie, a po zakończeniu widzi swój wynik, czas pracy oraz informację, czy przekroczył wymagany próg zaliczenia wynoszący 75%. Tryb praktyki to środowisko w stylu LeetCode z edytorem kodu Monaco, w którym uczeń rozwiązuje zadania programistyczne w wybranym języku (Python, C++, Java lub JavaScript), a jego kod jest automatycznie oceniany przez model Gemini według kryteriów egzaminacyjnych. Aplikacja śledzi ponadto postępy użytkownika — przechowuje historię sesji, statystyki per kategoria oraz serię dni nauki.

Projekt stawia na czytelność, automatyzację i naukę przez praktykę, a nie tylko na bierne czytanie teorii.

2. Technologie

Aplikacja ExamIQ zbudowana jest wyłącznie po stronie klienta — nie posiada własnego backendu ani bazy danych. Cały stos technologiczny opiera się na nowoczesnych standardach frontendowych.

React 18 + Vite stanowią fundament projektu. React odpowiada za deklaratywne budowanie interfejsu użytkownika w oparciu o komponenty funkcyjne i hooki, natomiast Vite pełni rolę narzędzia do budowania i serwowania aplikacji, zapewniając błyskawiczne odświeżanie widoku podczas developmentu (HMR) oraz zoptymalizowany bundle produkcyjny.

JavaScript (JSX) jest jedynym językiem programowania użytym w projekcie. Kod napisany jest zgodnie ze standardem ES2023 — wykorzystywane są m.in. destrukturyzacja, moduły ES, async/await oraz opcjonalne wiązanie. Składnia JSX umożliwia wygodne łączenie logiki komponentów z opisem struktury UI bezpośrednio w plikach .jsx.

CSS-in-JS (style inline) — warstwa wizualna aplikacji realizowana jest niemal w całości przez obiekty stylów przekazywane bezpośrednio do właściwości style komponentów React, uzupełnione lokalnie osadzonymi blokami <style> z definicjami animacji kluczowych (@keyframes). Takie podejście eliminuje zewnętrzne zależności od bibliotek stylów i zapewnia pełną enkapsulację wyglądu każdego komponentu.

LocalStorage API służy do trwałego przechowywania danych użytkownika po stronie przeglądarki. Zapisywana jest tam historia sesji egzaminacyjnych, statystyki nauki per kategoria oraz wyniki z trybu praktyki. Dane nie są nigdy wysyłane na serwer — użytkownik zachowuje pełną kontrolę nad swoją historią.

Cookie API przechowuje lżejsze dane sesyjne, takie jak nick użytkownika oraz licznik serii dni nauki (streak), które muszą być dostępne przy każdym odwiedzeniu aplikacji.

Plik questions.json oraz practice.json pełnią rolę lokalnej bazy danych z pytaniami. Pytania testowe i zadania programistyczne zakodowane są statycznie w formacie JSON i dołączane do aplikacji w trakcie budowania przez Vite.

Google Gemini API (@google/generative-ai) integruje model językowy bezpośrednio z aplikacją. Gemini używany jest dwutorowo: do oceny odpowiedzi na pytania otwarte w trybie nauki (porównanie merytoryczne zamiast dosłownego dopasowania) oraz do szczegółowej ewaluacji kodu ucznia w trybie praktyki według predefiniowanych kryteriów egzaminacyjnych.

Monaco Editor (edytor Visual Studio Code osadzony w przeglądarce) zapewnia pełnoprawne środowisko do pisania kodu w trybie praktyki — z podświetlaniem składni, autouzupełnianiem i numeracją linii.

React Router obsługuje nawigację między stronami aplikacji (trasy /learn, /exam, /about i inne) bez przeładowania strony, realizując model Single Page Application.

3. Opis głównych funkcjonalności

Poniżej opisano najważniejsze widoki i mechanizmy, które budują działanie aplikacji ExamIQ.

3.1 Strona główna (HomePage)

Strona główna pełni rolę punktu wejścia do aplikacji i prezentuje użytkownikowi wszystkie dostępne tryby oraz statystyki aplikacji. Sekcja Hero zawiera nagłówek, opis i dwa przyciski nawigacyjne. Dane statystyczne (liczba pytań, kategorii, próg zaliczenia) są zakodowane jako stała tablica STATS i renderowane przez komponent StatPill. Sekcja „Jak to działa?” prezentuje cztery kroki w układzie dwukolumnowym z wizualnym podglądem przykładowego pytania SQL — jest to statyczny mock demonstrujący wygląd quizu bez żadnej interakcji z danymi.

Kluczowy fragment
jsxconst STATS = [
  { value: '200+', label: 'pytań w bazie' },
  { value: '9',   label: 'kategorii tematycznych' },
  { value: '75%', label: 'próg zaliczenia' },
  { value: '∞',   label: 'prób egzaminacyjnych' },
];

Tablica STATS jest iterowana przez metodę .map() i każdy element przekazywany jest do komponentu StatPill przez spread operator {...s}. Komponent renderuje wartość i etykietę w stylizowanej karcie ze szklanym efektem (backdrop-filter: blur). Dzięki temu dane statystyczne są łatwe do zaktualizowania — wystarczy zmienić wartości w jednym miejscu, a zmiana automatycznie pojawi się we wszystkich kartach.

3.2 Tryb nauki — teoria (LearnPage)

LearnPage składa się z trzech faz zarządzanych przez zmienną stanu phase: konfiguracji (setup), quizu (quiz) i ekranu wyników (results). Przejścia między fazami realizowane są przez przekazywanie callbacków (onStart, onFinish, onRestart) między komponentami.

Konfiguracja quizu (SetupScreen)

jsxconst available = questions.filter(q =>
  typeMode === 'both' ? true : q.type === typeMode
).length;
const maxCount = Math.min(available, 40);

Zmienna available jest obliczana przy każdym renderze reaktywnie — filtruje tablicę pytań z pliku questions.json na podstawie aktualnie wybranego trybu (typeMode). Gdy użytkownik zmienia filtr na „zamknięte” lub „otwarte”, available automatycznie maleje, a suwak liczby pytań dostosowuje swój maksymalny zasięg do Math.min(available, 40). Zabezpiecza to przed sytuacją, w której użytkownik wybierze więcej pytań niż istnieje w bazie dla danego typu.

Losowanie pytań

jsxconst [pool] = useState(() => {
  const filtered = questions.filter(q =>
    config.typeMode === 'both' ? true : q.type === config.typeMode
  );
  return shuffle(filtered).slice(0, config.count);
});

Pula pytań jest inicjalizowana jednorazowo przy pierwszym renderze komponentu QuizScreen — dzięki leniwej inicjalizacji useState z funkcją inicjalizującą (zamiast wartości). Funkcja shuffle implementuje losowanie przez porównanie losowych wartości:

jsxfunction shuffle(arr) {
  return [...arr].sort(() => Math.random() - 0.5);
}

Spread operator [...arr] tworzy płytką kopię tablicy, nie modyfikując oryginału. Komparator () => Math.random() - 0.5 zwraca losowo wartości ujemne lub dodatnie, co powoduje losowe przestawienie elementów przez algorytm sortowania. Następnie .slice(0, config.count) przycina tablicę do wybranej liczby pytań.

Timer per pytanie

jsxuseEffect(() => {
  if (!config.timeLimit || revealed) return;
  setTimeLeft(config.timeLimit);
  timerRef.current = setInterval(() => {
    setTimeLeft(t => {
      if (t <= 1) { clearInterval(timerRef.current); handleReveal(false); return 0; }
      if (t <= 6) setTimerPulse(true);
      return t - 1;
    });
  }, 1000);
  return () => clearInterval(timerRef.current);
}, [idx, config.timeLimit, revealed, handleReveal]);

useEffect uruchamia się za każdym razem, gdy zmienia się numer pytania (idx) lub stan revealed. Warunek wyjścia if (!config.timeLimit || revealed) return sprawia, że timer nie startuje w trybie bez limitu ani po ujawnieniu odpowiedzi. Interwał odlicza sekundy przez aktualizator funkcyjny setTimeLeft(t => t - 1) — jest to bezpieczna forma zapisu, która gwarantuje odczytanie najnowszej wartości stanu, a nie stałej z zamknięcia (closure). Gdy t <= 6, ustawiana jest flaga timerPulse, która dodaje animację migania CSS. Gdy t <= 1, interwał jest czyszczony i wywoływana jest funkcja handleReveal(false) — pytanie zostaje zaliczone jako błędne. Funkcja zwrócona z useEffect (cleanup) czyści interwał przy odmontowaniu komponentu lub przy ponownym uruchomieniu efektu, zapobiegając wyciekom pamięci.

Ocena pytań otwartych przez Gemini

jsxconst checkWithGemini = async (question, correctAnswer, userAnswer) => {
  try {
    const ai = new GoogleGenerativeAI(import.meta.env.VITE_GEMINI_KEY);
    const model = ai.getGenerativeModel({ model: 'gemini-flash-latest' });
    const prompt = `Jesteś weryfikatorem odpowiedzi... Odpowiedz TYLKO jednym słowem: TAK lub NIE.`;
    const result = await model.generateContent(prompt);
    const text = result.response.text().trim().toUpperCase();
    return text.startsWith('TAK');
  } catch (err) {
    return userAnswer.trim().toLowerCase().includes(correctAnswer.trim().toLowerCase());
  }
};

Klucz API jest pobierany ze zmiennych środowiskowych Vite (import.meta.env.VITE_GEMINI_KEY), dzięki czemu nigdy nie trafia do repozytorium kodu. Prompt jest sformułowany tak, żeby model odpowiedział wyłącznie słowem „TAK” lub „NIE” — ogranicza to zużycie tokenów i eliminuje konieczność parsowania długiej odpowiedzi. Blok catch implementuje fallback — jeśli wywołanie API się nie powiedzie (brak internetu, wyczerpany limit), odpowiedź oceniana jest prostym porównaniem includes(). Nie jest to idealne, ale zapobiega całkowitemu zawieszeniu się funkcji oceniania.

Sekwencja ujawniania odpowiedzi

jsxconst handleReveal = useCallback((correct) => {
  clearInterval(timerRef.current);
  setRevealed(true);
  setIsCorrect(correct);
  if (correct) {
    setStreak(prev => { const n = prev + 1; if (n >= 2) setShowFlame(true); return n; });
  } else {
    setStreak(0);
    setShowFlame(false);
  }
  setResults(prev => [...prev, { q, correct }]);
}, [q]);

Funkcja opakowana w useCallback z zależnością [q] jest memoizowana — nie jest tworzona od nowa przy każdym renderze, co jest konieczne, ponieważ jest przekazywana jako zależność do useEffect timera. Wewnątrz funkcji najpierw czyszczony jest interwał timera, aby zapobiec podwójnemu wywołaniu. Streak (seria) jest aktualizowany przez funkcjonalny updater setStreak(prev => ...) — gwarantuje to odczyt aktualnej wartości w sytuacjach asynchronicznych. Wyniki są dodawane do tablicy immutowalnie przez spread [...prev, { q, correct }].

Zapis wyników

Ref saved.current pełni rolę flagi jednorazowego wykonania — bez niej useEffect z pustą tablicą zależności mógłby w trybie React Strict Mode uruchomić się dwukrotnie, co spowodowałoby podwójny zapis do historii. Wyniki per kategoria są mapowane do uproszczonej struktury { categoryName, wasCorrect }, którą hook useUserSession przetwarza na statystyki dashboardu.

3.3 Tryb praktyki — kodowanie (PracticePage)

PracticePage implementuje środowisko do pisania kodu wzorowane na LeetCode. Aplikacja zarządza trzema fazami: setup, editor i summary.

Integracja edytora Monaco

jsxuseEffect(() => {
  if (!window.monaco) { console.error('Monaco not loaded'); return; }
  if (monacoRef.current) return;

  monacoRef.current = window.monaco.editor.create(containerRef.current, {
    value: code,
    language: monacoLang,
    theme: 'vs-dark',
    fontSize: 14,
    fontFamily: "'JetBrains Mono', 'Fira Code', monospace",
    minimap: { enabled: false },
    automaticLayout: true,
  });

  monacoRef.current.onDidChangeModelContent(() => {
    setCode(monacoRef.current.getValue());
  });

  return () => {
    if (monacoRef.current) {
      monacoRef.current.dispose();
      monacoRef.current = null;
    }
  };
}, []);

Monaco Editor jest inicjalizowany jako zewnętrzna biblioteka ładowana przez window.monaco (nie przez npm), co pozwala uniknąć problemów z bundlowaniem. Ref monacoRef.current przechowuje instancję edytora poza cyklem życia Reacta — gdyby instancja była trzymana w stanie, każda zmiana kodu wywoływałaby ponowne renderowanie całego komponentu. Warunek if (monacoRef.current) return zapobiega tworzeniu duplikatów edytora przy ponownych renderach. Listener onDidChangeModelContent synchronizuje zawartość edytora ze stanem React code — dzięki temu kod jest dostępny podczas wysyłania do Gemini. Cleanup function wywołuje dispose(), które zwalnia zasoby DOM i wewnętrzne zasoby Monaco — bez tego przy odmontowaniu komponentu pozostałyby w pamięci osierocone instancje edytora.

Zmiana języka z potwierdzeniem

jsxfunction handleLangChange(newLang) {
  if (newLang === lang) return;
  const template = LANG_OPTIONS.find(l => l.id === newLang)?.comment || '';
  const isEmpty = code.trim() === '' || code.trim() === 
    (LANG_OPTIONS.find(l => l.id === lang)?.comment || '').trim();
  if (!isEmpty && !window.confirm('Zmiana języka wyczyści edytor. Kontynuować?')) return;
  setLang(newLang);
  setCode(template);
  if (monacoRef.current) {
    monacoRef.current.setValue(template);
    window.monaco.editor.setModelLanguage(model, monacoLang);
  }
}

Zmienna isEmpty sprawdza, czy użytkownik napisał już jakiś kod — porównuje aktualną zawartość z domyślnym szablonem języka. Jeśli edytor nie jest pusty, wyświetlane jest natywne okno potwierdzenia window.confirm. Zarówno stan React (setCode), jak i instancja Monaco (setValue) muszą być zaktualizowane oddzielnie — Monaco trzyma własny model danych niezależnie od React. Zmiana podświetlania składni wymaga wywołania setModelLanguage na modelu edytora.

Hook timera

jsxfunction useTimer(timerMode, onExpire) {
  const totalSeconds = timerMode === 'none' ? null : parseInt(timerMode) * 60;
  const [elapsed, setElapsed] = useState(0);
  const [running, setRunning] = useState(true);

  useEffect(() => {
    if (!running) return;
    intervalRef.current = setInterval(() => {
      setElapsed(prev => {
        const next = prev + 1;
        if (totalSeconds !== null && next >= totalSeconds) {
          clearInterval(intervalRef.current);
          setRunning(false);
          onExpire();
        }
        return next;
      });
    }, 1000);
    return () => clearInterval(intervalRef.current);
  }, [running, totalSeconds, onExpire]);

  return { remaining, elapsed, pct, fmt, danger, warning,
           pause: () => setRunning(false), resume: () => setRunning(true) };
}

Hook jest wydzielony jako osobna funkcja useTimer, co pozwala na jego reużycie i izolację logiki. Gdy timerMode === 'none', zmienna totalSeconds ustawiana jest na null, a timer pracuje jako stoper (zlicza czas w górę). Eksponowane funkcje pause i resume pozwalają komponentowi EditorScreen zatrzymać timer podczas oczekiwania na ocenę Gemini i wznowić go po zamknięciu panelu feedbacku. Właściwość danger (czas < 2 minuty) i warning (czas < 5 minut) służą do dynamicznej zmiany koloru timera w interfejsie.

3.4 Egzamin praktyczny z arkuszami (ExamPracticePage)

Strona arkuszy egzaminacyjnych umożliwia pracę z pełnymi zadaniami egzaminacyjnymi INF.04 i ocenę przesłanych plików przez AI.

Dropzone z obsługą plików i obrazów

jsxasync function handleEvaluate() {
  const genAI = new GoogleGenerativeAI(import.meta.env.VITE_GEMINI_KEY);
  const model = genAI.getGenerativeModel({ model: 'gemini-2.0-flash' });

  if (isImageFile(file)) {
    const b64 = await fileToBase64(file);
    parts = [
      { inlineData: { mimeType: file.type, data: b64 } },
      { text: `${systemPrompt}\n\nNa powyższym obrazie...` },
    ];
  } else {
    const text = await file.text();
    parts = [{ text: `${systemPrompt}\n\nKOD UCZNIA:\n\`\`\`\n${text.slice(0, 12000)}\n\`\`\`` }];
  }

  const res = await model.generateContent({ contents: [{ role: 'user', parts }] });
}

Funkcja isImageFile(file) sprawdza file.type.startsWith('image/'). Dla obrazów stosowany jest format inlineData z base64 — Gemini Vision przyjmuje obraz bezpośrednio w zapytaniu jako dane binarne, co pozwala modelowi ocenić wygląd aplikacji ze zrzutu ekranu. Dla plików tekstowych używana jest metoda file.text() (Web API), która odczytuje zawartość pliku jako string. Limit .slice(0, 12000) zapobiega przekroczeniu okna kontekstu modelu przy bardzo dużych plikach. Multimodalny format parts (tablica obiektów) pozwala łączyć obraz z tekstem w jednym żądaniu API.

jsxfunction fileToBase64(file) {
  return new Promise((resolve, reject) => {
    const r = new FileReader();
    r.onload  = () => resolve(r.result.split(',')[1]);
    r.onerror = () => reject(new Error('Nie można odczytać pliku'));
    r.readAsDataURL(file);
  });
}

FileReader.readAsDataURL zwraca string w formacie data:image/png;base64,AAAA.... Metoda .split(',')[1] wyodrębnia samą część base64, odcinając nagłówek MIME, który jest wymagany przez przeglądarkę, ale nieoczekiwany przez Gemini API.

Drag-and-drop

jsx<div
  onDragOver={e => { e.preventDefault(); setDragging(true); }}
  onDragLeave={() => setDragging(false)}
  onDrop={e => { e.preventDefault(); setDragging(false); handleFile(e.dataTransfer.files[0]); }}
  onClick={() => inputRef.current?.click()}
>
  <input ref={inputRef} type="file" style={{ display: 'none' }}
    accept=".py,.java,.cs,.cpp,.js,.jsx,.ts,.html,.css,.kt,.txt,.zip,image/*"
    onChange={e => handleFile(e.target.files[0])} />

e.preventDefault() w onDragOver jest konieczne — bez niego przeglądarka domyślnie otworzyłaby plik (np. wyświetliła obraz lub pobrała plik tekstowy). Zdarzenia onDragOver i onDragLeave zmieniają stan dragging, który wpływa na styl obramowania strefy upuszczania. e.dataTransfer.files[0] pobiera pierwszy plik z przeciągniętej kolekcji. Ukryty <input type="file"> jest wywoływany programatycznie przez inputRef.current?.click() po kliknięciu na strefę — to standardowy pattern do stylizowania inputów plikowych, ponieważ natywny element <input> jest trudny do ostylowania.

3.5 Dashboard i egzamin (DashboardPage)

Dashboard jest centralnym miejscem do śledzenia postępów, uruchamiania egzaminów i przeglądania historii sesji.

Losowanie puli egzaminacyjnej

jsxfunction startExam(count, category) {
  const base = category === 'all'
    ? ALL_QUESTIONS.slice()
    : ALL_QUESTIONS.filter(q => q.category === category);
  const pool = base
    .map(q => ({ q, r: Math.random() }))
    .sort((a, b) => a.r - b.r)
    .map(x => x.q)
    .slice(0, count);
  setExamPool(pool);
}

Jest to implementacja algorytmu tasowania metodą Fishera-Yatesa przez dołączanie wartości losowej — każdemu pytaniu przypisywana jest losowa liczba r, tablica jest sortowana po r, a następnie wartości losowe są odrzucane. Podejście to jest prostsze niż klasyczny Fisher-Yates, lecz nieco mniej równomiernie rozłożone statystycznie; dla potrzeb aplikacji edukacyjnej jest w zupełności wystarczające.

sessions.slice(-6) pobiera maksymalnie sześć ostatnich sesji. Wartość max jest obliczana przez Math.max(...last6.map(...), 1) — spread operator rozpakowuje tablicę do argumentów Math.max, a dodatkowe 1 zabezpiecza przed dzieleniem przez zero gdy wszystkie wyniki wynoszą 0. Wysokość każdego słupka jest obliczana jako pct * 0.5 (czyli połowa kontenera), a Math.max(..., 4) gwarantuje minimalną widoczną wysokość 4px nawet dla wyniku 0.
